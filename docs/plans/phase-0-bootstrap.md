## 5. Risk callouts

1. **Baileys multi-device policy + Linked-Device limit** ‚Äî WhatsApp caps concurrent linked devices (current published limit: 4). The brief implies a single bridge; if any team member has WhatsApp Web open, the QR auth may refuse. Plan: document a hard rule "only the bridge has the number linked" in `docs/ops.md` and surface in the runbook.
2. **Baileys session drift on disconnect** ‚Äî the socket can drop with no QR re-needed; brief #2 says "exponential backoff up to 60s, jitter, give up after 4 QR-needs." Risk: silent drops on flaky networks; we need a `last_message_at` staleness alarm in `/status` (#12 covers) and a Nagios-style check in the runbook.
3. **Composio `entity_id` quirks** ‚Äî `composio_outlook.py:84` auto-resolves from `user_id`; if `user_id` is missing on the connected account, we have a hard fail. Mitigation: `composio_outlook.py` already raises clearly; #2 must surface that error in `/status` so the runbook can name it.
4. **Composio `LIST_EVENTS` ignoring date filters** ‚Äî `composio_outlook.py:138-153` warns at ‚â•1000 events. Risk: edit/cancel lookup misses real events once the SAAC calendar passes 1000. Mitigation: when `BOOKING_HORIZON_DAYS=90` and events < 1000 it's safe; add a runtime check in #6 (`booking_edit.py`) that warns if the raw fetch returns 1000 and notify admins via `notify.py`.

6. **SQLite WAL + concurrent writers** ‚Äî `brief.md:80` says "SQLite WAL for concurrent reads." The orchestrator is a single writer; the only concurrent reader is `/health` and (eventually) a `repl`/admin shell. Plan: enable WAL via `PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL;` at boot, and **explicitly forbid** running two orchestrator processes against the same DB. 7. **LLM tool-calling schema not honoured by `INFERENCE_MODEL`** ‚Äî `MiniMax-M3` (the driver model in this session) is being used as the WhatsApp-side model too per `SKILL.md:33`. 8. **Phone filtering in the orchestrator** ‚Äî `brief.md:79` puts phone matching in the orchestrator. Risk: events matched by phone in body/attendees requires the body to actually contain the phone (in `+65‚Ä¶` form). If customers book through any other channel (form, email), those events are invisible to edit/cancel. Plan: log a warning every time the phone filter matches 0, so we can spot the gap.
9. **Booking horizon staleness** ‚Äî `BOOKING_HORIZON_DAYS=90` means edit/cancel lookups don't see tours beyond 3 months. Acceptable for v1; flag in runbook.
10. **NDJSON tail races the writer** ‚Äî `pygtail` handles `inode` changes (rotation), but if a line is written mid-write the reader sees a partial JSON line. Mitigation: use `json.loads` with try/except + skip-and-log, AND have wa-bridge write atomically (each line ends with `\n` after `fsync`). Add a `lsof` check in the runbook for `inbox.ndjson` not being open by another writer.

13. **`MiniMax-M3` as inference model** ‚Äî same model that writes code is also replying to customers. Risk: any drift in model behavior affects both. Plan: pin version explicitly in `INFERENCE_MODEL` and add a `--dry-run` flag to `router.py` that prints the would-be tool call without dispatching, for local testing.
